# Single-Responsibility File & Directory Layout

**Scope:** All source files in this project.

**Rule:** Organize code by responsibility, not by convenience. Each file and directory MUST have one clearly stated purpose, and file size MUST stay within strict limits to enforce that discipline.

**File size limits:**

- **Target:** ~120 lines per file (excluding blank lines and comments).
- **Hard ceiling:** 150 lines. Any file exceeding 150 lines MUST be split before merge.
- Generated code, lockfiles, and snapshot fixtures are exempt.

**Responsibility boundaries:**

1. **One concern per file.** A file exposes a single cohesive unit: one component, one hook, one service, one schema group, one route handler, one utility cluster. Unrelated helpers MUST live in separate files.
2. **One concern per directory.** Each directory represents a bounded capability (e.g., `auth/`, `billing/`, `user-profile/`). Cross-cutting utilities belong in a dedicated `shared/` or `lib/` tree, never scattered.
3. **Co-locate, don't centralize.** Tests, styles, types, and fixtures MUST sit next to the code they describe (`foo.ts`, `foo.test.ts`, `foo.types.ts`), not in distant global folders.
4. **Public surface via `index.ts`.** Each directory exposes its API through a single barrel file; consumers MUST import from the directory, not reach into internal files.
5. **No "utils.ts" dumping grounds.** Generic names like `utils`, `helpers`, `common`, `misc` are forbidden — name files after their actual responsibility.

**When a file approaches the limit:**

- Extract pure helpers into sibling files.
- Split a component into presentation + container, or into smaller sub-components.
- Move type definitions to `*.types.ts`.
- Promote a tightly-coupled file cluster into its own directory with an `index.ts`.
- Do NOT solve overflow by deleting comments, collapsing whitespace, or chaining expressions.

**Enforcement:**

- ESLint MUST enable `max-lines` (`{ max: 150, skipBlankLines: true, skipComments: true }`) and `max-lines-per-function` at a sensible threshold.
- CI MUST fail on any violation; `eslint-disable` for these rules is forbidden without an architectural justification recorded in the PR description.
- Reviewers MUST reject PRs that introduce catch-all files or directories, regardless of line count.

**Rationale:** Small, single-purpose files make ownership obvious, diffs reviewable, refactors safe, and merge conflicts rare. Line limits are a forcing function for clean decomposition, not an aesthetic preference.
